# LVGL 单片机移植指南：表情系统与动效实现规范

本文档旨在指导嵌入式工程师，如何在资源受限的单片机（特指 **ESP32-S3-WROOM-N16R8**）上，基于 LVGL 图形库高保真地还原整套表情物理动效系统。

## 1. 硬件平台与性能约束 (ESP32-S3-WROOM-N16R8)

- **硬件特性**：双核 240MHz，自带 16MB Flash 和 8MB PSRAM，支持 RGB/SPI 屏幕接口，具备一定的 2D 图形硬件加速（如 ESP-LCD 等）。
- **内存与资源池策略**：
  - 得益于 8MB PSRAM 的加持，项目依赖的所有矢量资源（如 PNG/SVG 图片、星星高光、泪滴、蓝牙/WiFi 等 Icon 图标）均可统一收拢于 `assets/` 文件夹中。
  - 在嵌入式移植时，建议将这些资源统一转码为 C 数组格式（通过 LVGL 官方图片转换工具），或利用 LVGL 的外部文件系统功能预加载至 PSRAM 中，避免运行时实时解码。
- **性能取舍 (30\~60 FPS 流畅运行原则)**：
  - **禁用高算力滤镜**：彻底移除 Web 端的实时高斯模糊 (`filter: blur`) 和动态阴影 (`filter: drop-shadow`)。
  - **基于矩阵变换的物理模拟**：所有生命力表现（软弹感、重量感、果冻效应）均通过纯粹的**缩放 (Scale)**、**旋转 (Rotation)**、**平移 (Translation)** 和**透明度 (Opacity)** 的精确时间轴与数学函数组合来实现。
  - **层级架构**：需严格管理 Z-Order。大面积图层渲染时（如全屏眼白和遮罩），建议开启 LVGL 的双缓冲（Double Buffering）以避免画面撕裂。

## 3. 表情状态机架构设计 (播放与打断机制)

在底层图形渲染时，所有的表情切换必须使用 **平滑过渡（Lerp 线性插值）**，绝不能瞬间跳帧。
为避免多事件并发时的冲突与错乱，系统首先区分了 **底态 (Base State)** 和 **瞬态 (Transient)** 两个核心概念：

- **底态**：系统无限循环的基础状态。只有 `100+ 日常发呆` 和 `300+ 音乐律动`（播放音乐期间）属于底态。
- **瞬态**：打断底态的临时状态（如触摸反馈、系统指令）。瞬态结束后必须交还控制权。

在此基础上，状态机遵循以下三大核心机制：

### 3.1 抢占优先级 (Priority)

高优事件无条件抢占低优，低优事件被高优拦截。优先级从低到高：

- **Level 0 (底态)**：最低优的无限循环基础状态。两者之间互斥，不发生暴力抢占，而是平滑过渡。
  - **触发情况**：设备处于闲置无操作状态；或处于长期播放音乐/音频的背景状态。
  - **对应表情**：闲置时对应 `100+ Daily` 系列（如 `101 扫视`, `103 困倦`）；播放音乐时对应 `300+ Music` 系列（如 `301 经典`, `302 螺旋`）。

- **Level 1 (物理触摸)**：物理交互的瞬态或维持态反馈。
  - **触发情况**：用户通过物理方式与设备发生短暂或持续的互动，如轻触、拍打、抚摸、长按、摇晃设备等。
  - **对应表情**：对应 `200+ Touch` 系列（如瞬态的 `201 纳尼`, `208 弹弹`，或长按维持的 `204 坠入爱河`, `206 委屈`）。

- **Level 2 (语音交互)**：由于包含复杂且连续的上下文，中断体验代价极高，因此优先级高于单次触摸。
  - **触发情况**：用户通过唤醒词触发人机对话，包含设备被唤醒、正在聆听用户说话、识别思考以及语音播报回复的完整生命周期。
  - **对应表情**：对应 `500+ Hear` 系列（如 `501 唤醒`, `502 聆听`, `503 收到`, `504 疑惑`）。

- **Level 3 (系统指令)**：最高优先级！必须无条件抢占屏幕，无论当前用户是在触摸还是在语音对话。
  - **触发情况**：设备发生系统级的核心状态变更或执行强制任务，如开关机、网络状态变更（蓝牙/WiFi搜索与断开报警）、用户强按物理按键调节音量、媒体控制等。
  - **对应表情**：对应 `400+ Command` 系列（如开关机 `401/402`，网络反馈 `405~410`，音量调节 `411/412`）。

### 3.2 切入机制 (打断与覆盖)

当收到新事件时：

1. **被拦截**：若新事件优先级 **<** 当前优先级，请求被忽略或挂起排队。
2. **强制抢占**：若新事件优先级 **≥** 当前优先级，立即淡入新表情。
   - ⚠️ **防错关键**：抢占时必须 **清除旧表情的倒计时定时器**，防止旧定时器到期后强行切回导致状态错乱。

### 3.3 切出机制 (恢复与退场)

当高优的瞬态表情（如触摸、语音）结束时，需平滑淡出并 **恢复到被打断前的底态**（即 `100+ 日常` 或 `300+ 音乐`）。
退场方式分为两类：

1. **倒计时自动切出 (定时动画)**
   - **适用**：固定时长的瞬态表情（如 `201 纳尼`、`401 开机`、`503 收到` 等的大部分瞬态表情）。
   - **规则**：状态机开启一个倒计时（如 3000ms），时间一到自动退场。
2. **事件被动切出 (维持状态)**
   - **适用**：时长未知的等待/长按表情（如 `405 蓝牙搜索`、`204 长按触摸`、`502 聆听` 等维持态表情）。
   - **规则**：不设倒计时，无限循环播放。直到收到 **明确的外部结果**（如：松手、蓝牙连接成功、语音识别结束）才触发退场。

## 4. 动效类型与参数拆解

为了在 LVGL 中完美复刻，需要深刻理解各种形变与特效的代码原理。**建议工程师按以下物理特性分批开发与测试**：

### 4.1 矩阵变换核心 (平移、缩放、旋转)

- **形变与缩放主导 (Squash & Stretch)**：用于表现肌肉张力与果冻感。
  - **表情归类**：`103 困倦`, `105 疲劳`, `117 虚脱`；`202 摇摇`, `209 愤怒`, `210 大的来了`；`301 经典`, `305 弹簧`；`501 唤醒`。
  - **LVGL 对应 API**：`lv_img_set_zoom`。
- **旋转主导 (Rotation)**：
  - **表情归类**：`207 眩晕`；`302 螺旋`, `308 旋转`；`504 疑惑`。
  - **LVGL 对应 API**：`lv_img_set_angle`。
- **平移与运动轨迹主导 (Translation)**：
  - *极坐标追踪*：随机生成极坐标半径与角度平滑移动。**归类**：`101 扫视`, `102 跟踪`, `111 扫描`, `116 偷看`。
  - *数学曲线 (sin/cos)*：平滑圆周或钟摆运动。**归类**：`115 催眠`, `118 大葱明`, `121 苍蝇飞飞`；`206 委屈`；`309 摇摆`。
  - *物理弹簧与反弹*：硬碰撞与衰减震荡 (`exp(-t) * cos(t)`)。**归类**：`120 乒乒乓乓`；`208 弹弹`, `211 蛮王冲撞`；`503 收到`。

### 4.2 引入图层与特殊表情 (Shape Classes)

此类表情超越了单纯的黑底眼珠，需提前准备额外的 `lv_img` 资源并做好 Z-Order 管理，**建议打包开发**：

- **特殊瞳孔特效 (附加动画层)**：
  - **表情归类**：`122 眼泪汪汪` (泪滴 SVG/PNG), `205 星星眼` (四角星), `204 坠入爱河` (波浪线), `300+ 音乐系列` (心形等 Emoji)。
  - **说明**：叠加了独立的高光图层，拥有自己独立的缩放、旋转、透明度动画，必须挂载在眼珠容器下同步移动。
- **悬浮 UI 图标层 (Floating Icon)**：
  - **表情归类**：`405/406/407 蓝牙系列`、`408/409/410 WiFi系列`。
  - **说明**：WiFi/蓝牙等图标悬浮于双眼之间，拥有独立的 `icon-x/y` 运动轨迹和渐变逻辑。
- **特殊动效滤镜降级**：
  - **表情归类**：`303 拖尾`、`306 溶解`。
  - **说明**：在 Web 中依赖高算力 `filter: blur`。在 ESP32 上需改为多图层透明度叠加（Motion Blur）或多帧图片预渲染序列帧。

### 4.3 物理边界限制 (Circular Constraint)

为防止眼珠飞出物理 TFT 屏幕的边缘，所有的坐标更新逻辑末尾，都必须加入“勾股定理”圆周约束。

- **一般模式 (最大半径 33px)**：适用 90% 以上的基础表情。
- **受限模式 (最大半径 15px)**：**表情归类**：`122 眼泪汪汪`、`205 星星眼`（因瞳孔形变极其巨大，需缩小活动范围防止切边）。

### 4.4 视差与延迟跟随 (Parallax & Delay)

- **眼白视差 (Sclera Parallax)**：眼白的移动幅度是黑眼珠移动幅度的 `0.22` \~ `0.4` 倍，营造球体包裹的 3D 立体感。
- **非对称神经延迟**：右眼的动作（如眨眼闭合、眼皮下垂）比左眼滞后约 `40ms`，打破机器人的呆板同步感。
- **眼白形变滞后**：在 `300+ 经典音乐` 模式下，眼白的物理形变（挤压/撑开）严格滞后于内部 Emoji 图层约 `70ms` (2% 进度)。

### 4.5 眨眼系统接管 (Blink Logic)

- **全局定时器**：随机眨眼（Blink）作为影响上下眼睑 Y 轴的最高优图层，独立运行。带有贝塞尔曲线模拟肌肉粘滞感。
- **强制拦截规则**：在 `300+` 音乐律动期间、以及 `502 聆听`、`105 疲劳` 等部分高度依赖瞳孔张力的表情播放时，**必须从代码层面彻底屏蔽随机眨眼**，防止其强制形变破坏当前表情的视觉核心。

## 5. LVGL 迁移核心方法论建议

1. **架构设计：逻辑运算与渲染分离 (双任务模型)**
   - 建立一个频率较高（如 50Hz）的 **逻辑定时器 (Logic Task)**，纯粹运行数学运算：根据 `time`、`currentMode` 计算出当前帧左/右眼的 X, Y, Scale, Rotation, Opacity，以及眼白和附属图标的坐标。
   - 在 LVGL 的主循环（`lv_timer_handler`）中，仅负责将上述计算好的纯数值，通过 `lv_obj_set_pos`、`lv_img_set_zoom`、`lv_img_set_angle` 应用到各个图层控件上。
2. **避免内存碎片化与频繁重绘**
   - 在系统初始化阶段，一次性创建好所有可能用到的 `lv_img` 图层（眼眶背景、眼白裁剪层、左眼珠、右眼珠、星星高光 1/2/3、悬浮图标层、音乐 Emoji 层）。
   - 运行时**绝对不要**频繁调用 `lv_obj_create` 或删除对象。而是通过 `lv_obj_add_flag(obj, LV_OBJ_FLAG_HIDDEN)` 或调整 `lv_obj_set_style_img_opa` 来控制各图层的显隐。
3. **合理选用** **`lv_anim_timeline_t`** **(时间轴)**
   - 对于高度固定、没有随机变量的循环动画（如 `300+ 音乐系列`，`401 开机自检`，`503 收到` 划对勾），极其推荐使用 LVGL 的时间轴 API。它能完美还原 Web CSS `@keyframes` 中的百分比控制，省去大量 C 语言的 `if-else`。
   - 对于带有随机目标点、实时追踪特性的动画（如 `101 扫视`，`405 蓝牙追踪`），则必须在 C 代码中使用数学公式实时演算插值（Lerp）。
4. **浮点数与数学库优化**
   - 虽然 ESP32-S3 带有浮点运算单元 (FPU)，但大量的 `sin()`、`cos()`、`exp()` 仍可能带来负担。
   - 在不影响视觉精度的情况下，建议将部分高频计算替换为查表法（LVGL 内置了 `lv_trigo_sin/cos` 等快速查表函数），或对 Web 源码中的极小尾数进行定点数优化。
